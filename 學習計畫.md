# CI/CD 實戰學習計畫

> 總時長：約 3 小時（含休息）
> 對象：有基本 Docker 經驗的初學者
> 前置條件：已完成 Docker 教育訓練

---

## 課程目標

- 理解 CI/CD 的概念與為什麼需要它
- 認識 GitHub Actions 的運作方式（Workflow / Job / Step / Action）
- 能撰寫基本與多 Job 的 workflow 檔案
- 實作兩個完整部署流程：
  - **Demo A**：Next.js 靜態網站 → GitHub Pages（無需 Docker）
  - **Demo B**：Next.js + Docker Image → Docker Hub → Railway 雲端部署

## 兩個 Demo 對照

| 面向 | Demo A：GitHub Pages | Demo B：Railway |
|------|---------------------|-----------------|
| 專案位置 | [hello-world/](hello-world/) | [hello-world-railway/](hello-world-railway/) |
| Next.js 模式 | `output: 'export'`（靜態） | `output: 'standalone'`（Server） |
| 容器化 | 不需要 | Dockerfile（三階段 Build） |
| Job 數 | 1 個（deploy） | 3 個（test → build-push → deploy） |
| 需要的 Secrets | 0 個（用 `GITHUB_TOKEN`） | 3 個（Docker Hub 帳密 + Railway Webhook） |
| 部署目標 | GitHub Pages `gh-pages` branch | Railway（拉 Docker Hub image） |
| 成品網址 | `{帳號}.github.io/hello-world/` | `your-app.railway.app` |
| 適合場景 | 靜態頁面、文件、部落格 | 有後端邏輯的 Web App |

## 面試重點整理

| 主題 | 常見問題 |
|------|----------|
| CI/CD 概念 | CI 和 CD 的差異？為什麼需要自動化部署？ |
| GitHub Actions | Workflow、Job、Step、Action 的關係？ |
| GitHub Actions | 什麼是 Runner？self-hosted runner 是什麼？ |
| GitHub Actions | `needs` 關鍵字的用途？Job 之間如何依賴？ |
| 實務應用 | 如何安全管理密鑰（Secrets）？為何不該用密碼而用 Token？ |
| 實務應用 | Docker Image 的 Tag 策略：`latest` vs git SHA vs 語意版本 |
| 實務應用 | 靜態部署 vs 容器化部署的取捨？ |
| 實務應用 | 如何讓外部服務（如 Railway）收到新版本通知？ |

## 延伸學習

| 主題 | 說明 |
|------|------|
| GitLab CI / Jenkins | 其他常見 CI/CD 平台，概念相通、語法不同 |
| ArgoCD | GitOps 風格的 CD 工具（搭配 Kubernetes） |
| GitHub Actions Marketplace | 社群提供的現成 Action |
| Matrix / Cache / Artifacts | GitHub Actions 進階功能 |
| Branch Protection Rule | 在 merge 前強制要求 CI 通過 |
| 其他部署目標 | AWS ECS、GCP Cloud Run、Vercel、Fly.io 等 |
