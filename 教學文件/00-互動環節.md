## 互動 1：快問快答（5 min）

> **時機**：在 Demo 1（第一個 Workflow）之後

### 題目 1（暖身題）
**Q：CI 和 CD 最大的差別是什麼？**

<details>
<summary>答案</summary>

- **CI（持續整合）**：專注在「程式碼合併後自動測試 / 建置」，確保品質
- **CD（持續交付 / 部署）**：專注在「測試通過後自動打包和部署」，加速上線

簡單記法：CI 管「品質」，CD 管「速度」。

</details>

---

### 題目 2
**Q：GitHub Actions 的 Workflow 檔案要放在哪裡？**

選項：
- A. 專案根目錄
- B. `.github/actions/`
- C. `.github/workflows/`
- D. `workflows/`

<details>
<summary>答案</summary>

**C**。Workflow YAML 檔案必須放在 `.github/workflows/` 目錄下，GitHub 才會自動偵測並執行。

</details>

---

### 題目 3
**Q：在 GitHub Actions 的 Workflow 中，`runs-on: ubuntu-latest` 是什麼意思？**

<details>
<summary>答案</summary>

指定這個 Job 要在 **GitHub 提供的最新版 Ubuntu Linux 機器**上執行。這台機器叫做 **Runner**，每次執行都是**全新的環境**（用完即丟）。

也可以選 `windows-latest` 或 `macos-latest`，或自架的 self-hosted runner。

</details>

---

### 題目 4（加分題）
**Q：`uses: actions/checkout@v4` 在做什麼？如果不加這一步會怎樣？**

<details>
<summary>答案</summary>

`actions/checkout@v4` 會把你的 repo 的程式碼 clone 到 Runner 上。

如果不加這一步，Runner 上是**完全空白**的——沒有你的程式碼、沒有 `package.json`、沒有 Dockerfile，什麼都沒有。後面的 `npm install`、`npm run build`、`docker build` 全部會失敗。

**所以幾乎每個 Workflow 的第一步都是 checkout。**

</details>

## 互動 2：情境題挑戰（5 min）

> **時機**：在 Demo A（GitHub Pages）之後

### 題目 1
**Q：Demo A 的 workflow 有使用 `${{ secrets.GITHUB_TOKEN }}`，但我們並沒有在 Settings 裡面設定這個 Secret。為什麼還能用？**

<details>
<summary>答案</summary>

`GITHUB_TOKEN` 是 GitHub 在每次 Workflow 執行時**自動產生**的暫時性 Token，不需要自己設定。

它的權限由 workflow 裡的 `permissions:` 控制：
```yaml
permissions:
  contents: write   # 允許推到 gh-pages branch
```

- 執行完 Workflow 這個 Token 就失效了
- 只在當前 repo 有效
- 所以用它來推 `gh-pages` branch 既方便又安全

</details>

---

### 題目 2
**Q：Demo A 的 `next.config.mjs` 為什麼要設定 `basePath: '/hello-world'`？不設會怎樣？**

<details>
<summary>答案</summary>

因為 GitHub Pages 的網址不是根目錄，而是：
```
https://{帳號}.github.io/{repo名稱}/
```

如果不設 `basePath`，Next.js 會以為網站掛在根目錄 `/`，產生的所有連結和資源路徑都會錯誤——頁面可能載入，但 CSS、JS、圖片全部 404。

設了 `basePath: '/hello-world'` 之後，所有路徑會自動加上 `/hello-world` 前綴，才能對應到 GitHub Pages 的實際網址。

</details>

### 題目 3
**情境：你的同事改了一行 code，PR 的 CI 測試失敗了。他說：「這只是小改動，不用管測試，直接 merge 就好。」你應該怎麼回應？**

<details>
<summary>答案</summary>

**不應該直接 merge！**

理由：
1. CI 存在的意義就是在合併前抓出問題
2. 「小改動」也可能造成大問題——這正是自動化測試要幫你發現的
3. 正確做法：先修好測試失敗的問題，再 merge

**最佳實踐**：在 GitHub 設定 **Branch Protection Rule**，要求 CI 通過才能 merge，讓制度強制執行而不是靠人盯。

</details>

## 互動 3：實戰判斷題（5 min）

> **時機**：在 Demo B（Railway 完整 Pipeline）之後

### 題目 1
**Q：你的 CI/CD Pipeline 每次 push 都會 build 並 push Docker image，tag 都是 `latest`。這樣有什麼問題？**

<details>
<summary>答案</summary>

問題：
1. **無法追蹤版本**：不知道 `latest` 對應的是哪次 commit
2. **無法回滾**：想回到上一個版本，但 `latest` 已經被覆蓋
3. **快取問題**：有些環境會快取 `latest`，不會自動拉新版

解決方案（Demo B 就是這樣做）：
```yaml
tags: |
  ${{ secrets.DOCKERHUB_USERNAME }}/hello-world-railway:latest
  ${{ secrets.DOCKERHUB_USERNAME }}/hello-world-railway:${{ github.sha }}
```

同時打兩個 tag：`latest`（方便取用）+ git SHA（可追蹤、可回滾）。

</details>

### 題目 2
**Q：Demo B 的 workflow 有三個 Job 依 `needs` 串起來：**

```
test → docker-build-push → deploy-to-railway
```

**如果 `docker-build-push` 失敗了，`deploy-to-railway` 會執行嗎？**

<details>
<summary>答案</summary>

**不會！** 因為 `deploy-to-railway` 有 `needs: docker-build-push`，表示它依賴前一個 Job。

`docker-build-push` 失敗後，`deploy-to-railway` 會自動跳過（顯示灰色 "Skipped"）。

這就是 Pipeline 的安全機制：
- Image 沒推上 Docker Hub → Railway 拉不到新版本 → 不應該觸發部署
- 前面壞了，後面就不要做

</details>

---

### 題目 3
**Q：Demo B 的三個 Job 各自在「獨立的 Runner」上執行。也就是說 `test` 在 A 機器、`docker-build-push` 在 B 機器、`deploy-to-railway` 在 C 機器。那麼 Job 之間怎麼傳遞檔案？**

選項：
- A. 檔案會自動複製到下一個 Job
- B. 下一個 Job 要自己 `checkout` 或從 artifact 下載
- C. 透過 `/tmp` 共用
- D. 第一個 Job 跑完後永遠保留機器

<details>
<summary>答案</summary>

**B**。每個 Job 都在**獨立、全新的機器**上跑，彼此之間不共享檔案系統。

所以你會看到 Demo B 的每個 Job 都各自做：
```yaml
- uses: actions/checkout@v4
```

如果真的需要跨 Job 傳遞檔案，要用：
- `actions/upload-artifact` + `actions/download-artifact`
- 或 `actions/cache`

> 但我們的 Demo B 中，每個 Job 做的事都是獨立的：test 只需要 source、docker-build-push 也只需要 source、deploy 只 curl Webhook——所以各自 checkout 就夠了。

</details>

### 題目 4
**Q：以下哪種做法最安全？你的 CI 需要登入 Docker Hub。**

- A. 把帳號密碼直接寫在 workflow YAML 裡
- B. 把帳號密碼寫在 `.env` 檔裡，然後 commit 到 repo
- C. 用 GitHub Secrets 儲存 Access Token
- D. 寫在 README 裡讓大家都能用

<details>
<summary>答案</summary>

**C**。用 GitHub Secrets + **Access Token（不是密碼）**！

- A 和 B 都會把憑證暴露在 repo 裡（git 歷史記錄永遠都在）
- D 不用說了 😂
- Secrets 的值不會出現在 log 中（會被遮成 `***`），也不會被 fork 的 repo 看到
- 用 **Access Token** 而不是密碼：Token 可以隨時撤銷、權限可以限縮（Read-only / Read & Write）

</details>
