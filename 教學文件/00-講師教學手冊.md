> 總時長：約 3 小時（含休息）
> 對象：已完成 Docker 教育訓練的學員

> 範例專案：[hello-world/](../hello-world/)（Demo A）、[hello-world-railway/](../hello-world-railway/)（Demo B）

## 課前準備清單

- [ ] GitHub 帳號（講師用來現場 Demo）

- [ ] Railway 帳號（用 GitHub 登入即可）
- [ ] 預先建好兩個 Demo 用的 repo：
  - `hello-world`（Next.js 靜態輸出 → GitHub Pages）
  - `hello-world-railway`（Next.js + Docker → Railway）
- [ ] 在 `hello-world-railway` repo 中設定 Secrets：
  - `DOCKERHUB_USERNAME`

  - `RAILWAY_WEBHOOK_URL`
- [ ] 確認兩個 repo 的 GitHub Actions 事先都跑過一次成功
- [ ] 在 Railway 預先建立「Deploy from Docker Hub image」專案
- [ ] 如果學員要跟著做，提前通知準備 GitHub / Docker Hub / Railway 帳號
- [ ] 預先開好需要的網頁分頁：
  - 兩個 repo 的 Actions 頁面
  - Docker Hub 的 repository 頁面
  - Railway 專案頁面
  - GitHub Secrets 設定頁面

### 1. 開場（10 min）

**目標**：承接 Docker 課程，說明「有了 Docker 之後的下一步」。

**過場引言**：
> 「上一堂課我們學了 Docker，可以把應用打包成 Image，用 Compose 管理多容器。但問題來了——每次改 code 都要手動 build、手動 push、手動部署... 能不能讓這些事情全自動？這就是今天要學的 CI/CD。今天我們會做兩個真實的部署：一個靜態網站上 GitHub Pages，一個容器化應用上 Railway。」

**講解重點**：
1. 回顧 Docker 的價值：打包一次、到處執行
2. 痛點：手動 build + push + deploy 太慢、太容易出錯
3. CI/CD 的核心價值：**自動化整條交付流程**
4. 今天的兩個里程碑：
   - Demo A：Push → 自動 build → 部署到 GitHub Pages（免 Docker）
   - Demo B：Push → 測試 → Build Docker Image → Push Hub → Railway 重新部署

**參考教材**：[01-CI-CD基礎概念.md](01-CI-CD基礎概念.md)

---

### 2. 第一段：CI/CD 核心概念（20 min）

**目標**：建立 CI/CD Pipeline 的心智模型。

**講解重點**：

**(a) CI vs CD（8 min）**
- CI = 持續整合：每次 push code → 自動跑測試 / 建置
- CD = 持續交付 / 部署：測試通過 → 自動準備 / 部署
- 畫出 Pipeline 流水線圖（Source → Build → Test → Package → Deploy）

**(b) Pipeline 各階段與今日對照（7 min）**
- 對照今天的兩個 Demo，說明每個階段會對應到什麼：
  - Source：GitHub push event
  - Build：`npm run build`
  - Test：`npm test` / build 編譯
  - Package：靜態檔輸出 / Docker image
  - Deploy：gh-pages branch / Railway

**(c) GitHub Actions 基本概念（5 min）**
- Workflow、Job、Step、Action 的關係
- 用食譜類比（Workflow = 食譜、Job = 一道菜、Step = 步驟、Action = 預製調味料）

**參考教材**：[01-CI-CD基礎概念.md](01-CI-CD基礎概念.md)、[02-GitHub-Actions基礎.md](02-GitHub-Actions基礎.md)

### 3. Demo 1：第一個 Workflow（15 min）

**目標**：讓學員看到「push code 後自動跑東西」的效果。
**使用 repo**：任一空白 repo 或 [hello-world/](../hello-world/) 的前置狀態。

**步驟**：

```bash
mkdir -p .github/workflows
```

建立 `.github/workflows/hello.yml`：

```yaml
name: Hello CI

on: [push]

jobs:
  say-hello:
    runs-on: ubuntu-latest
    steps:
      - name: Say Hello
        run: echo "Hello, CI/CD!"

      - name: Show date
        run: date

      - name: Show runner info
        run: |
          echo "OS: $(uname -s)"
          echo "User: $(whoami)"
          echo "Directory: $(pwd)"
```

第二輪：加入 `actions/checkout@v4`，示範「沒 checkout 就是空機器」。

**講解要點**：
- 「這個 YAML 就是在告訴 GitHub：每次 push，在一台乾淨 Ubuntu 機器上跑這些指令」
- 切到 GitHub Actions 頁面即時看結果
- 強調 Runner 每次都是全新環境（用完即丟）
- `uses` vs `run` 的差別

### 5. 第二段：GitHub Actions 進階（20 min）

**過場引言**：
> 「剛剛 echo 只是熱身。接下來我們要做真正的部署——先從靜態網站上 GitHub Pages 開始。」

**講解重點**：

**(a) 多 Job 與 needs（7 min）**
- 平行 vs 依序執行
- `needs` 關鍵字
- 「先跑測試，測試通過才部署」

**(b) Secrets 與 Permissions（7 min）**
- 為什麼不能把密碼寫在 YAML 裡
- 兩種權限模型：
  - 內建 `GITHUB_TOKEN`（Demo A 會用到）— 免設定，只要 `permissions` 宣告
  - 使用者自訂 Secrets（Demo B 會用到）— 手動到 Settings 設定
- `${{ secrets.XXX }}` 語法

**(c) 條件與觸發（6 min）**
- `on: push` vs `pull_request`
- `workflow_dispatch` — 手動觸發按鈕
- `if:` 條件（PR 只測試、push 才部署）

**參考教材**：[02-GitHub-Actions基礎.md](02-GitHub-Actions基礎.md)

### 6. Demo A：Next.js → GitHub Pages（20 min）

**目標**：示範無需 Docker、無需額外 Secrets 的最簡單部署流程。
**使用 repo**：[hello-world/](../hello-world/)

**步驟**：

1. **展示專案結構**（2 min）：`app/page.js`、`next.config.mjs`（注意 `output: 'export'` 和 `basePath`）
2. **說明 workflow**（5 min）：逐行講 [hello-world/.github/workflows/deploy.yml](../hello-world/.github/workflows/deploy.yml)
3. **現場修改 page.js，push**（5 min）：
   ```bash
   git add . && git commit -m "Update greeting" && git push
   ```
4. **觀察 Actions 執行**（3 min）：5 個 Step 依序跑完
5. **打開成品網址**（3 min）：`https://{帳號}.github.io/hello-world/`
6. **講解 GitHub Pages 設定**（2 min）：Settings → Pages 選 `gh-pages` branch

**講解要點**：
- 靜態輸出為什麼要 `basePath` —— GitHub Pages 網址不是根目錄
- `permissions: contents: write` 才能推到 `gh-pages` branch
- `peaceiris/actions-gh-pages` 幫你把 `out/` 推到 `gh-pages`
- **整個流程 0 個 Secrets**，因為用內建 `GITHUB_TOKEN`

**參考教材**：[04-NextJS-GitHub-Pages.md](04-NextJS-GitHub-Pages.md)

### 8. 第三段：Docker 整合 GitHub Actions（15 min）

**過場引言**：
> 「靜態網站很簡單，但真實專案通常需要一個會跑的 Server。這時候就要搬出 Docker——把應用打包成 Image，再讓雲端平台拉去跑。」

**講解重點**：

**(a) 為什麼要 Docker Hub（5 min）**
- 容器化部署流程：Build → Push to Registry → Pull & Run
- Docker Hub 是最常見的 public registry
- Access Token 比密碼安全，且可撤銷

**(b) 官方 Docker Actions（5 min）**
- `docker/login-action@v3`
- `docker/build-push-action@v5`
- 為什麼比 `docker build` + `docker push` shell 指令更好（支援 buildx、快取、多平台）

**(c) Image Tag 策略 + Deploy Hook（5 min）**
- `latest` vs git SHA vs 語意版本
- 為什麼正式環境不該只用 `latest`
- Deploy Hook：通知外部平台「有新版本了」的機制（Railway、Render、Fly.io 都有類似概念）

**參考教材**：[03-Docker整合GitHub-Actions.md](03-Docker整合GitHub-Actions.md)

### 9. Demo B：完整 CI/CD Pipeline → Railway（25 min）

**目標**：串起完整流程 push → test → build → push image → Railway 部署。
**使用 repo**：[hello-world-railway/](../hello-world-railway/)
**這是整堂課的高潮！**

**步驟**：

1. **展示專案結構**（3 min）：[Dockerfile](../hello-world-railway/Dockerfile)（三階段）、[next.config.mjs](../hello-world-railway/next.config.mjs)（`output: 'standalone'`）
2. **講解 3 個 Job**（7 min）：打開 [deploy.yml](../hello-world-railway/.github/workflows/deploy.yml)
   - Job 1 `test`：`npm run build` 當作測試
   - Job 2 `docker-build-push`：`needs: test`，登入並推兩個 tag
   - Job 3 `deploy-to-railway`：`needs: docker-build-push`，`curl` Webhook
3. **展示 Secrets 設定**（3 min）：`DOCKERHUB_USERNAME` / `DOCKERHUB_TOKEN` / `RAILWAY_WEBHOOK_URL`
4. **修改 page.js 並 push**（3 min）
5. **Actions 即時觀察**（5 min）：三個 Job 依序亮綠燈、Job 依賴關係圖
6. **驗證結果**（4 min）：
   - Docker Hub 上看到 `latest` 和 `{sha}` 兩個 tag
   - Railway 儀表板顯示「Deploying」→「Active」
   - 打開 `https://your-app.railway.app` 看到新版本

**講解要點**：
- 從頭走一遍：「我只是 push code，後面 test、build、push image、部署全部自動」
- 故意讓一個 step 失敗，展示失敗 Job 後續被自動 skip 的效果
- 對比 Demo A：多了容器化、多了 Secrets、多了 Deploy Hook，但本質都是 `.github/workflows/*.yml`

**參考教材**：[05-Railway-Docker-Deploy.md](05-Railway-Docker-Deploy.md)

## 講師備忘

### 節奏控制

- 每 20 分鐘穿插一次互動或 Demo，避免純講太久
- Demo A 跑很快（1-2 分鐘），Demo B 比較久（3-5 分鐘），抓時機講下一段
- 準備「已經跑完」的備用截圖，以防 Actions 排隊或網路問題

### 常見學員問題

| 問題 | 回答方向 |
|------|---------|
| 「GitHub Actions 要錢嗎？」 | 公開 repo 免費，私有 repo 每月 2000 分鐘免費 |
| 「Railway 要錢嗎？」 | 有免費額度，超過要升級 |
| 「為什麼不直接從 GitHub 部署到 Railway，要繞 Docker Hub？」 | Demo 目的是展示完整 CI + image registry 流程；實務上 Railway 也支援直連 GitHub |
| 「公司用 GitLab 怎麼辦？」 | 概念相通，GitLab CI 語法不同但邏輯一樣 |
| 「可以部署到 AWS 嗎？」 | 可以，但超出今天範圍，可以自行延伸學習 |

### 緊急備案

- **Actions 卡住**：先講下一段，回來再看
- **Docker Hub push 失敗**：檢查 Token 權限是 Read & Write，備案只 demo 到 `docker build`
- **Railway Webhook 沒反應**：確認 URL 正確，備案用 Railway 儀表板手動 Redeploy
